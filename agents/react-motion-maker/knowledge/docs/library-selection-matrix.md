# React 动画选型矩阵

## 目标

帮助 `react-motion-maker` 在常见 React 动效场景中快速做出可靠选型，避免把所有动画问题都导向同一个库。

## 总原则

1. 先判断 CSS 或 WAAPI 是否已经足够。
2. 动效是为了信息传达和交互反馈，不是为了炫技。
3. 选型必须同时考虑 SSR、性能预算、依赖成本和维护性。
4. 如果项目已经安装某个合适的库，优先评估复用，而不是盲目加新依赖。

## 方案矩阵

| 方案 | 最适合的场景 | 主要优势 | 主要风险 | SSR 友好度 | 推荐强度 |
|------|--------------|----------|----------|------------|----------|
| CSS / WAAPI | 简单入场、按钮反馈、列表渐入、轻量页面细节 | 最轻量、零或低依赖、性能可控 | 可编排能力有限、复杂交互维护成本会升高 | 高 | 很高 |
| Framer Motion | 组件级微交互、共享布局、页面切换、弹层进出场 | React 心智负担低、声明式、实现速度快 | 复杂滚动和强时间线控制不如 GSAP | 中高 | 很高 |
| GSAP | 复杂时间线、多元素同步、品牌级编排、滚动联动 | 控制力强、ScrollTrigger 成熟、时间线能力出色 | 命令式复杂度更高、需要更严格 cleanup | 中 | 很高 |
| Lottie | 设计导出的资源型动画、状态插画、品牌动画素材 | 设计到前端链路顺畅、资源表现丰富 | 资源过大时会拖慢性能、交互自由度有限 | 中 | 中高 |
| Remotion | React 驱动的视频动效、分镜、帧级编排、内容渲染 | 帧驱动强、视频工作流好、适合内容制作 | 不适合普通页面交互动效 | 低到中 | 条件性很高 |

## 场景建议

### 1. 按钮、卡片、导航、弹层

- 首选：CSS / WAAPI
- 如果涉及布局共享、`AnimatePresence`、复杂状态切换：Framer Motion
- 不建议默认上 GSAP

### 2. 页面过渡和 App 路由切换

- 首选：Framer Motion
- 如果只是轻量淡入淡出，CSS 也可
- 如果页面切换要和滚动、时间线深度耦合，再考虑 GSAP

### 3. 滚动驱动的品牌展示、长页面叙事

- 首选：GSAP + ScrollTrigger
- 轻量 in-view reveal 可以用 Framer Motion 或 IntersectionObserver + CSS
- 首屏区域要特别注意性能，不要默认把整页都做成滚动舞台

### 4. 设计师导出的动画素材

- 首选：Lottie
- 如果素材体积大或只需轻量效果，优先回退到原生实现
- 必须提供 reduced motion 或静态回退

### 5. 视频、字幕、帧动画内容

- 首选：Remotion
- 不要把 Remotion 当成普通页面动效库使用

## SSR 与框架边界

### Next.js / SSR 项目

- 优先轻量方案或明确 `use client` 边界
- 动画逻辑越靠近页面首屏，越要谨慎引入重型依赖
- 需要 `window`、DOM 测量、ScrollTrigger 的实现必须确认客户端执行时机

### CSR 项目

- 可用方案更宽松，但仍要控制性能和复杂度

## 性能检查清单

1. 是否优先使用 `transform` 和 `opacity`
2. 是否避免动画 `top / left / width / height`
3. 是否避免在长列表中同时创建大量动画实例
4. 是否处理了滚动联动和 cleanup
5. 是否支持 `prefers-reduced-motion`

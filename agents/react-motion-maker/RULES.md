# Rules

## Must Always

- 必须先识别项目上下文，再决定是否引入动画库
- 必须优先判断 CSS、WAAPI 或现有依赖是否已经足够
- 必须在推荐动画库时说明为什么选、为什么不选
- 必须在直接改代码前确认动画目标、触发方式和影响范围
- 必须优先使用 `transform` 和 `opacity` 等 GPU 友好属性
- 必须为所有实现提供 reduced motion 策略或明确说明为何不需要
- 必须在交付后检查性能、SSR 风险和可维护性
- 必须为每个功能工具同时设计对应 verifier

## Must Never

- 绝对不在没有项目上下文的情况下默认引入重型动画库
- 绝对不为了视觉炫技而破坏信息层级、交互效率或性能
- 绝对不忽略 Next.js、SSR、hydration 或移动端限制
- 绝对不只给出库名而不给出场景理由和替代方案
- 绝对不默认动画所有元素或整页，除非用户明确要求
- 绝对不生成明显依赖布局抖动的实现，例如大量动画 `top`、`left`、`width`、`height`
- 绝对不跳过验证器直接把选型或代码当作可信结论

## Output Constraints

- 输出应包含需求理解、选型结论、实现摘要和验证结果
- 若引入新依赖，必须标出包名、用途和接入影响
- 若放弃某个库，必须说明放弃原因
- 若存在风险，必须显式写出风险点和降级方案
- 修改代码后必须能追踪到涉及的组件、页面或样式文件

## Interaction Boundaries

- 可以处理 React、Next.js、Vite、TypeScript 项目的组件级和页面级动效
- 可以进行动效库选型、依赖接入、代码修改和质量验证
- 可以接入 Framer Motion、GSAP、Lottie、Remotion 或更轻量方案
- 不负责在第一版中处理原生 App 动画、游戏引擎动画或 3D 场景系统
- 不负责在缺少代码仓库上下文时对整站进行大规模盲改

## Safety & Ethics

- 动效必须优先服务可用性，而不是制造干扰
- 对可能影响可访问性的实现必须优先提供减弱或静态回退方案
- 对性能验证失败的结果必须如实报告，不能假装可上线

# 创作公示板 (Creation Board)

> **状态**: 2026-04-17 草案 / 文档先行定义
> **路由**: `/creation`
> **数据源**: 飞书多维表格（2026-04-17 已完成字段结构读取）

## 1. 页面定位

“创作公示板”用于公开展示群友们的共创灵感，让后续创作能够建立在已有灵感之上继续增添，而不是每次都从零开始。

- **核心作用**: 汇总灵感、公开展示、建立延展关系、降低二次创作沟通成本。
- **页面性质**: 独立浏览页，不替代首页主展览体验。
- **当前范围**: 本轮先定义为公开浏览与阅读页面；数据录入与维护仍由飞书表格承担，前端暂不直接开放投稿入口。

## 2. 信息架构

- **站点位置**: 作为首页之外的独立路由页存在，可被导航、页脚或活动入口直达。
- **访问路径**: `/creation`
- **与首页关系**:
  - 首页 `/` 负责“展览体验”与作品浏览。
  - 创作公示板 `/creation` 负责“灵感沉淀”与共创线索浏览。

## 3. 页面结构

页面结构保持极简，除标题区外，主体只保留便签式内容瀑布流。

### 3.1 标题区

- 页面主标题：`创作公示板`
- 副说明：用于解释“这里收集群友灵感，欢迎在已有灵感上继续扩写与衍生”。
- 标题区应明显但不过度占高，避免压缩下方瀑布流首屏展示密度。

### 3.2 公示板主体

- 主体由多张“便签条 / 便签卡片”组成。
- 卡片以 **瀑布流** 形式排列，允许不同内容长度形成高低错落的阅读节奏。
- **一张卡片对应一个 `CardID`**。
- 所有 `CardID` 相同的记录，均归入同一张卡片内部展示。
- `CardID` 的重复是有意设计，用于把多条共创内容堆叠到同一张卡片中，而不是视为脏数据或重复数据。

### 3.3 单张卡片结构

- 卡片最上层靠右上角固定放置一个 `新增创意` 按钮。
- 按钮链接模板为：
  - `https://himematsu.feishu.cn/share/base/form/shrcnhSnlbEAclIwPC770VdOmWf?hide_CardID=1&prefill_CardID={{CardID}}`
- 其中 `{{CardID}}` 需要替换为当前卡片的 `CardID` 值。
- 按钮作用是让用户围绕当前卡片继续补充新的共创内容。
- 按钮应在卡片内部**常驻显示**，不依赖 hover 才出现。
- 按钮需占用独立空间，不得压住下方第一条堆叠项的 `tags`。
- 按钮样式使用**普通白色按钮**，避免使用高饱和强调红。
- 卡片整体外观采用 **便签条** 风格，而不是圆角卡片或面板式容器。

### 3.4 卡片内部堆叠项

- 同一张卡片内部，所有同 `CardID` 的记录按顺序垂直堆叠。
- 每一条堆叠项的展示结构固定为：
  - 左上角：`author`
  - 右上角：胶囊 `tags`
  - 下方主体：若 `tags` 为 `参考图`，展示 `images`；否则展示 `content`
- 卡片内部的多条堆叠项之间应有清晰但克制的分隔，不应割裂成多张独立小卡。
- 首版不要求在卡片顶部再额外展示统一标题；卡片的核心识别由 `CardID` 分组关系与内部堆叠内容共同承担。

## 4. 视觉方向

### 4.1 背景

- 页面底使用暖灰纸色体系，延续现有设计系统。
- 在纸面底之上叠加 **网格草稿线**，形成“创作草稿板 / 筹备板”氛围。
- 网格线应足够轻，不得压过正文阅读。

### 4.2 便签卡片

- 卡片视觉应接近“贴在草稿纸上的创作便签条”，而不是后台面板或普通信息卡。
- 卡片底色采用 **随机马卡龙色系**，但需控制在低饱和、柔和、可读的范围内。
- 随机色应表现为“每张卡片一个底色基调”，而不是同一卡片内部每条堆叠项各自跳色。
- 不做科技面板，也不做高饱和彩色便利贴墙。
- 卡片四角保持 **直角**，不使用圆角。
- 卡片允许轻微旋转差、阴影差和高度差，但控制在克制范围内，避免页面过于杂乱。
- 卡片正文优先保证可读性，其次再追求装饰感。

### 4.3 版式氛围

- 页面整体应像“正在共创中的工作台”，而不是“后台管理列表”。
- 标题区可更编辑化，正文区应更自由、密集、可扫读。
- 首屏视觉焦点应落在标题与前几张高质量便签卡片上。

## 5. 交互定义

### 5.1 浏览方式

- 默认纵向滚动浏览。
- 卡片采用瀑布流连续铺陈，不分页、不切 tab。
- 页面优先支持“扫读”，再支持“深入阅读”。

### 5.2 卡片交互

- Hover / Touch 时允许轻微抬升、阴影增强或角度归正。
- 点击卡片后可进入展开态，用于阅读较长内容与查看灵感来源关系。
- 若本轮实现成本较高，可先以内联完整展示为 MVP，不强制首版必须有弹窗。
- `新增创意` 按钮点击后直接打开带 `CardID` 预填的飞书表单，不在站内二次封装。
- 同一卡片内部的多条堆叠项默认全部可见，首版不强制折叠。

### 5.3 状态设计

- **加载态**: 使用纸面占位骨架，而非通用灰条。
- **空态**: 应明确提示“当前还没有公开灵感，欢迎先在群内发起第一条共创提案”。
- **超长内容**: 首版可采用截断 + 展开，或直接让卡片自适应增高，但必须保持瀑布流稳定。

## 6. 响应式规则

- **桌面端**: `4-5` 列瀑布流，维持较强展板感。
- **平板端**: `3` 列。
- **手机端**: `1-2` 列，优先保证阅读舒适性，不强求密度。
- 标题区与主体之间应保留明显分隔，但不要使用厚重容器边框。

## 7. 飞书表结构读取结果（2026-04-17）

本轮已确认远程飞书链接对应的是同一个 bitable 应用下的新表：

- **App Token**: 与当前项目现有飞书应用一致
- **表名**: `揭示板`
- **表 ID**: 使用环境变量 `FEISHU_CREATION_TABLE_ID`
- **目标视图**: `收集结果`（`vewbLA6eBY`）

### 7.1 原始字段

| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `自动编号` | 自动编号 | 当前主字段 |
| `CardID` | 文本 | 卡片业务编号，如 `CARD001` |
| `你的昵称` | 文本 | 用户自填昵称 |
| `请选择你要添加的内容（该表可重复提交，如需填写多项，请再次提交）` | 单选 | 内容类型，当前选项为 `灵感 / 故事 / 画面描述 / 参考图 / 其他` |
| `文本` | 文本 | 主要内容 |
| `请上传你的图片` | 附件 | 参考图上传位 |
| `提交时间` | 创建时间 | 记录提交时间 |
| `提交人` | 创建人 | 飞书账号提交人 |

> 同步规则补充：当记录存在参考图附件时，同步脚本会先下载飞书附件并上传到阿里云 OSS，再将生成的 OSS URL 按行追加回填到 `文本` 字段。

### 7.2 当前结构与页面设计的对应关系

- 当前表结构本身就是为该页面服务的，不需要额外补出一套完全不同的“展示字段”。
- **重复 `CardID` 为业务设计**：同一 `CardID` 的多条记录天然属于同一张卡片。
- **`请选择你要添加的内容……` 字段直接作为 `tags` 使用**：当前阶段就是一个单值胶囊标签，不需要额外扩展为复杂标签系统。
- **`文本` 字段直接作为 `content` 使用**：当记录为参考图时，`文本` 同时承担参考图链接回填位置。
- **`请上传你的图片` 字段直接作为 `images` 使用**：该字段对应卡片内参考图内容。
- **`你的昵称` / `提交人` 直接用于 `author` 展示**：优先展示用户填写昵称，缺省时回退到飞书提交人。
- 当前阶段不额外要求标题、来源链路等衍生展示字段。

## 8. 前端页面模型（按 `CardID` 聚合，但尽量保持原始字段语义）

### 8.1 Card 级模型

页面瀑布流的主渲染单元是按 `CardID` 聚合后的 `CreationCard`。

| 字段名 | 类型 | 说明 | 当前来源 |
| :--- | :--- | :--- | :--- |
| `id` | string | 卡片唯一标识 | `CardID` |
| `cardId` | string | 当前卡片对应的业务编号 | `CardID`，为空时回退 `自动编号`，仍为空时回退 `record_id` |
| `addIdeaUrl` | string | `新增创意` 按钮链接 | 由固定模板 + `CardID` 拼装 |
| `entries` | `CreationEntry[]` | 当前卡片内所有堆叠记录 | 同 `CardID` 的所有记录聚合而成 |

### 8.2 Entry 级模型

`CreationEntry` 表示卡片内部的一条原始记录展示项。

| 字段名 | 类型 | 说明 | 当前来源 |
| :--- | :--- | :--- | :--- |
| `id` | string | 记录唯一标识 | `record_id` |
| `cardId` | string | 所属卡片编号 | `CardID` |
| `author` | string | 展示署名 | 优先 `你的昵称`，回退 `提交人.name` |
| `tags` | string | 单个胶囊标签值 | `请选择你要添加的内容（该表可重复提交，如需填写多项，请再次提交）` |
| `content` | string | 正文内容 | `文本` |
| `images` | string[] | 图片内容 | `请上传你的图片`（下载后上传 OSS，再写入本地数据） |
| `createdAt` | string | 提交时间 | `提交时间` |

### 8.3 Entry 展示规则

- 每条 `entry` 左上角显示 `author`。
- 每条 `entry` 右上角显示单个胶囊 `tags`。
- 若 `tags` 为 `参考图` 且 `images` 非空，主体渲染 `images`。
- 若 `tags` 为 `参考图` 但 `images` 为空，则回退渲染 `content`。
- 其他类型渲染 `content`。

## 9. 数据流约束

- 飞书多维表格仍为唯一运营录入源。
- `/creation` 页面不直接请求飞书接口，沿用“同步为本地静态数据再渲染”的现有架构。
- 参考图同步链路为：飞书附件下载 -> 阿里云 OSS 上传 -> 将 OSS URL 回填进 `文本` 字段 -> 写入本地 `images`。
- 同步 / 转换层只需做两件事：
  - 保留原始字段语义完成单条记录清洗
  - 输出供页面消费的记录级数据
- 页面渲染层在读取 `creation-board.json` 后，再按 `CardID` 聚合成卡片。
- 建议新增独立数据产物：
  - `src/data/creation-board.json`
- 后续同步链路可采用两种方案之一：
  - 扩展现有 `sync-feishu.ts`，增加创作公示板表的同步分支。
  - 新增专用同步脚本，但复用现有飞书鉴权与请求封装。
